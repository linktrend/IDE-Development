# Blocked States

## Purpose

This document defines blocked conditions and the evidence required when work cannot proceed.

## Universal Block Rule

Blocked means work cannot validly proceed now.

Blocked does not mean:

- unfinished
- difficult
- uncertain
- undesired

Blocked means a real progression condition is unsatisfied.

## Valid Block Causes

Valid block causes include:

- unsatisfied dependency
- missing required input
- unresolved decision
- insufficient proof
- failed review awaiting rework
- failed integration awaiting correction
- ambiguous state or ownership

## Work Unit Block Conditions

### Intent

Intent is blocked when:

- a go or no-go decision cannot be made
- required business, product, or scope information is missing

### Program

Program is blocked when:

- modules cannot be defined coherently
- program constraints are unresolved
- release expectations are materially ambiguous

### Module

Module is blocked when:

- it cannot be reduced to executable issues
- required child work cannot become ready
- required module review cannot proceed

### Phase

Phase is blocked when:

- none of its required issues can proceed
- phase entry or exit conditions remain unsatisfied

### Issue

Issue is blocked when:

- dependency satisfaction is absent
- scope cannot be executed safely
- review or integration failure prevents progression

### Proof

Proof is blocked when:

- required evidence cannot be collected
- proof remains materially insufficient

### Review

Review is blocked when:

- subject or proof is too ambiguous to evaluate
- mandatory criteria are missing

### Integration

Integration is blocked when:

- review has not passed
- integration risks or conflicts prevent safe incorporation

## Required Block Trail

Every blocked state must preserve:

- the reason
- attempted progression
- unresolved conditions
- next required resolution path
