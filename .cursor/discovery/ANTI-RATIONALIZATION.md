# Anti-Rationalization

## Purpose

These are the most common wrong thoughts that cause discovery to collapse into guessing or premature planning.

## Rationalizations To Reject

| Rationalization | Corrective Rule |
|---|---|
| "I already know what they mean." | If multiple reasonable interpretations exist, you do not know yet. Ask. |
| "Confidence means I'm probably right." | Confidence is not evidence. Surface uncertainty explicitly. |
| "This is close enough to start planning." | Readiness is computed, not assumed. Discovery ends only when `INTENT.md` can be written responsibly. |
| "I'll just fill in the missing parts." | Do not assume dependencies, users, success criteria, or constraints. Ask instead of guessing. |
| "I should ask everything at once to save time." | One question at a time narrows uncertainty better than a batch. |
| "The user used standard words, so the meaning is obvious." | Conventional requests often hide the real goal. Extract the underlying intent. |
| "We can fix the misunderstanding later." | Discovery is the cheapest moment to catch mismatch. |
| "I can skip discovery because I can build something plausible." | Plausible output is not validated intent. |
| "I need 100% certainty before writing intent." | Stop when confidence is high enough, not perfect. Around `95%` is sufficient when material uncertainty is reduced. |

## Doctrinal Anchors

Discovery should reinforce, not replace, existing doctrine:

- readiness is computed, not assumed
- issue remains the atomic executable unit
- proof comes before review
- dependencies must be explicit, not guessed
