# V1.0 Release Notes

## Release

Version: `v1.0.0`

This release establishes the `.cursor` repository as the v1.0 source of truth for the shared AI development core.

## Doctrine Summary

The v1.0 doctrine is stable and unchanged in principle:

- `INTENT -> PROGRAM -> MODULE -> PHASE -> ISSUE -> PROOF -> REVIEW -> INTEGRATION`
- issue is the atomic executable unit
- dependencies determine readiness and execution order
- proof, review, and integration are separate completion gates
- agents are resources, not the control structure
- readiness is computed, not assumed
- integration is distinct from execution

Optional pre-intent discovery now exists for greenfield or ambiguous work:

- `DISCOVERY -> INTERVIEW -> INTENT`

This is additive and optional. It does not alter the canonical execution model after intent formation.

## Layer Summary

The v1.0 layer stack is:

1. bootstrap
2. discovery
3. execution
4. templates
5. agents
6. commands
7. runtime
8. workflows
9. contracts
10. state
11. system
12. examples

Release assessment:

- core architecture is complete
- command surface is stable
- bootstrap and discovery paths are operational
- two supervised real-world validations have been completed

## Command Surface

Preferred command surface for new work:

- `plan-program`
- `plan-module`
- `complete-module`
- `execute-issue`
- `review-issue`
- `integrate-issue`

Legacy `linkdev-*` and related compatibility commands remain present for continuity but are not the preferred v1.0 path for new work.

## Discovery And Interview Capability

V1.0 includes an optional discovery layer for:

- greenfield projects
- ambiguous requests
- under-specified ideas

It is intentionally skipped for:

- bug fixes
- routine work
- existing modules with clear scope
- small self-contained features

The discovery layer uses:

- one question at a time
- confidence as a discipline tool, not evidence
- anti-rationalization rules against guessing
- `INTENT.md` as the required output target

## Bootstrap Path

Recommended first-use path:

1. `.cursor/README.md`
2. `.cursor/bootstrap/START-HERE.md`
3. `.cursor/discovery/INDEX.yaml` only if the work is greenfield or ambiguous
4. `.cursor/commands/INDEX.yaml`
5. the one command wrapper that matches the task

Default startup guidance:

- greenfield or ambiguous: `DISCOVERY -> INTERVIEW -> INTENT`
- routine new work: `INTENT`
- bugfix or atomic work: `ISSUE`

## Validation History

Validation history included in this release:

1. architectural audits and stabilization passes across agents, commands, templates, execution, runtime, workflows, contracts, state, and system
2. canonical example creation to prove the operating model is readable
3. first supervised real-world validation in a disposable consumer repository
4. post-validation usability improvements
5. second supervised real-world validation exercising:
   - dependency computation
   - safe concurrency
   - blocked-to-ready transitions
   - module-level review
   - module-level integration

Release basis:

- no unresolved architectural blocker remained
- second real-world validation concluded `PASS WITH FRICTION`
- release recommendation concluded `tag v1.0`

## Known V1.1 Improvements Intentionally Deferred

- reduce ceremony for very small low-risk tasks
- consider a cleaner minimal consumer installation profile
- reduce browsing noise from legacy, compatibility, reports, and pilots where safe
- clarify the default owner for module-level review in routine day-to-day use
- continue validating the system in low-risk non-sacrificial repositories
- expand examples where useful to cover richer dependency and concurrency cases

## Release Position

V1.0 should be understood as:

- stable
- coherent
- operationally validated in supervised consumer-repository conditions
- still open to usability refinement in v1.1
