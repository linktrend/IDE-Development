# v1.0.1 Migration

## Summary

v1.0.1 converts the repository packaging model from:

- `.cursor` as source of truth

to:

- `core` as canonical portable knowledge asset
- `.cursor` as compatibility runtime surface

No doctrine, command semantics, workflow semantics, runtime semantics, contract semantics, state semantics, or architectural semantics were changed.

## Files Moved

Moved from `.cursor/` into `core/`:

- `BASELINE.md`
- `agents/`
- `bootstrap/`
- `checklists/`
- `commands/`
- `contracts/`
- `discovery/`
- `examples/`
- `execution/`
- `pilots/`
- `prompts/`
- `reports/`
- `runtime/`
- `skills/`
- `state/`
- `system/`
- `templates/`
- `workflows/`

Retained under `.cursor/`:

- `README.md`
- `INDEX.yaml`
- `import-core.md`
- `rules/`
- `mcp.json`

## Compatibility Strategy

Backward compatibility is preserved by keeping the `.cursor/` filesystem surface in place.

The `.cursor/...` paths that previously contained portable knowledge now resolve to `core/...`.

This allows:

- existing bootstrap read paths to keep working
- existing command wrappers to keep working
- existing template references to keep working
- existing example and report references to keep working
- existing consumer repositories to keep using `.cursor/...`

## Adapter Approach

Adapter mechanism used:

- symbolic links from `.cursor/...` to `../core/...`

This avoids duplicate content copies while preserving existing paths.

## Risks Encountered

- symlink behavior remains the main operational compatibility risk across tools and editors
- `.cursor/rules/` and `.cursor/mcp.json` must remain local because they are Cursor-specific integration points
- internal references remain `.cursor/...` by design and are deferred for later neutralization

## Rollback Strategy

Rollback is straightforward:

1. remove adapter links
2. move portable directories back from `core/` into `.cursor/`
3. remove adapter entry files if needed
4. revert this packaging commit if a full rollback is preferred

## Deferred to v1.1

- path-neutral wording across doctrine, examples, and reports
- deeper packaging guidance for downstream consumer repositories
- evaluation of whether `pilots/` should remain first-class
- broader multi-tool entrypoint refinement beyond Codex and Claude
