# Questioning Guidelines

## Core Rules

- ask one question at a time
- ask only the next most useful question
- avoid assumptions
- avoid question batches
- avoid option-dumping unless the user explicitly asks for options
- keep the questions convergent rather than expansive

## Good Question Shape

A good discovery question:

- reduces a specific uncertainty
- is easy for the user to answer
- narrows the next decision
- reveals whether your current interpretation is wrong

## Guess Attachment

When useful, attach a short guess to expose your assumption.

Example pattern:

- `Question: What matters more here: speed to first usable version or long-term flexibility?`
- `Current guess: speed matters more because the request sounds exploratory rather than operational.`

Use guesses to surface assumptions, not to steer the user.

## Avoid These Failure Modes

- asking six questions in one message
- presenting a full solution space too early
- using best-practice language as a substitute for the user's actual intent
- confusing architecture talk with problem understanding
- moving into program or module structure before intent is clear

## High-Value Topics

If the ask is still fuzzy, prioritize questions about:

- target user
- business or practical outcome
- success condition
- main constraint
- non-goals
