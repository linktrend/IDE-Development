# Retrospective Learning

Use this skill after significant work, repeated blockers, failed reviews, or completed modules.

## Use When

- a module completes
- a review finds repeated issues
- an issue was blocked repeatedly
- a workflow felt too heavy or unclear
- a bug escaped verification
- a skill or command should be improved

## Reflection Questions

- What worked?
- What failed or slowed the work?
- Which blockers were genuine?
- Which proof was useful?
- Which tests or checks were missing?
- Should any command, skill, template, or artifact be updated?

## Rules

- Convert lessons into actionable changes or explicit follow-ups.
- Do not rewrite doctrine based on one anecdote unless the failure is structural.
- Preserve compatibility when improving workflows.
- Separate local project lessons from shared core lessons.

## Output

- concise retrospective summary
- lessons learned
- recommended system updates
- follow-up issues or reports
- no-change decision if no update is justified

## Progressive Disclosure

Read completed proof, review, integration, blockers, and handoff reports for the target scope. Do not scan unrelated historical reports unless pattern evidence is needed.
